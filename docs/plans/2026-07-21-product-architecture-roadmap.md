# StudyMate 下一阶段产品与技术开发计划

> 日期：2026-07-21  
> 适用范围：现有 TypeScript CLI MVP 到“考试调研 + 自适应学习 + 拟人化搭子”的下一阶段  
> 计划性质：建议稿；用于统一 PRD、架构和开发优先级，不代表现有功能已经完成

## 1. 结论先行

StudyMate 下一阶段不应继续堆叠孤立命令，而应围绕一个明确的产品对象——`ExamProject`（一次考试备考项目）——重组为三个闭环：

1. **开考前建档闭环**：考试建档 → 搜索备考经验与官方资料 → 来源确认 → 资料入库 → 知识结构化 → 生成计划。
2. **每日学习闭环**：查看今日任务 → 阅读材料 → 完成任务 → 针对当日知识点出题 → 批改反馈。
3. **自适应闭环**：积累掌握度与错误证据 → 识别薄弱点 → 插入或调整复习任务 → 调整后续出题。

“备考搭子”不是第四条独立业务线，而是覆盖三个闭环的交互层。它应解释系统决策、陪伴用户执行并感知情境，而不是只在命令末尾说一句话。

当前文件驱动架构可以保留，但需要纠正两处表述：

- `workspace/event_log/events.jsonl` 目前是**审计日志**，不是可重放的单一事实来源；在实现 reducer、幂等处理和状态重建之前，不应宣称完整 Event Sourcing/Replay。
- Obsidian 适合作为资料与产物浏览器，但无法单独承载“实体化搭子”的主体验。CLI 保留为开发和高级入口，产品主界面应逐步转向本地 Web UI；Obsidian 作为可选导出/浏览能力。

## 2. PRD 校准

### 2.1 产品定位

**建议的一句话定位：**

> StudyMate 是一个以考试目标为中心的个人备考 Agent：它先帮用户弄清“考什么、别人怎么准备、资料从哪里来”，再把可信资料变成可执行计划，通过每日学习和测验持续调整复习节奏，并由一个可选择的拟人化搭子陪伴执行。

### 2.2 目标用户

下一阶段只服务一个主用户：

- 有明确考试名称和考试日期；
- 备考周期约 2 周至 6 个月；
- 单用户、单考试项目优先；
- 有若干本地资料，但不知道如何筛选、拆解和安排；
- 愿意每天完成 20～120 分钟的学习任务。

暂不同时解决学校作业辅导、开放式自学、多人班级管理和培训机构后台。这些场景会迫使数据模型和权限系统过早复杂化。

### 2.3 核心价值顺序

1. **少走弯路**：优先找到官方范围和高质量备考经验。
2. **知道今天做什么**：把长期考试目标转为现实可执行的日计划。
3. **知道自己哪里不会**：测验、错题和掌握度形成可追踪证据。
4. **有人陪着做下去**：搭子提升执行感和情绪体验，但不替代学习决策的可解释性。

### 2.4 下一阶段范围

#### 必须实现

- 考试项目建档：考试名称、日期、目标、基础、每日时间。
- 备考调研：官方来源、经验来源、资料推荐，保留引用和采集时间。
- 来源确认：用户确认后才能进入知识库，防止低质量经验污染知识点。
- 多资料安全导入、去重、稳定 ID 和来源追踪。
- 结构化切片、概念依赖、循环检测和来源回链。
- 覆盖完整备考周期的计划，不再限制为 14 天。
- 基于当日任务和历史薄弱点出题。
- 累积式错误画像、掌握度历史和可插入任务的动态调整。
- 可选择的搭子角色、跨会话记忆、关键节点解释和陪伴反馈。
- 全流程可审计；核心决策能说明依据。

#### 明确不做

- Agent 自动修改自己的源代码或未经评估自动改 Prompt。
- 多用户、班级和机构后台。
- 扫描 PDF OCR、视频转写、复杂公式识别。
- IRT、FSRS、主观题自动评分作为首阶段硬要求。
- 3D 角色、实时动画、完整语音陪伴。
- 一开始就拆微服务、接数据库或构建插件平台。

### 2.5 “自进化”的产品定义

为避免概念膨胀，将能力分成三层：

| 层级 | 定义 | 下一阶段 |
|---|---|---|
| 用户自适应 | 根据完成情况、正确率和掌握度调整计划与题目 | 必须实现 |
| 策略优化 | 比较不同复习间隔、题目策略和干预方式的效果 | 记录数据，后续灰度 |
| 系统自修改 | 自动改代码、Prompt 或架构并直接上线 | 不做 |

任何策略优化都必须经过“候选策略 → 离线评估 → 小范围试用 → 指标比较 → 可回滚发布”，不能把不可控的自修改包装成自进化。

## 3. 关键用户流程

### 3.1 首次建档

```text
输入考试名称、日期、目标与每日时间
  → Agent 生成考试检索计划
  → 收集官方大纲、报名/考试规则、经验帖、教材与真题线索
  → 按来源类型和可信度整理
  → 用户确认考试画像与资料清单
  → 下载或导入资料
  → 结构化与知识点拆解
  → 用户确认首版计划
```

调研产物建议：

```text
workspace/exam.json
workspace/research/sources.jsonl
workspace/research/exam_profile.json
workspace/research/experience_insights.md
workspace/research/material_recommendations.md
workspace/research/approved_sources.json
```

### 3.2 每日学习

```text
搭子说明今天为什么安排这些任务
  → 用户开始/跳过/调整任务
  → 查看对应知识切片和来源
  → 标记完成
  → 生成只覆盖今日内容和到期薄弱点的测验
  → 提交答案并查看证据化解析
```

### 3.3 批改与调整

```text
批改
  → 按知识点累计本次证据
  → 区分未掌握、记忆模糊、审题/粗心等错误
  → 更新掌握度历史和薄弱点画像
  → 必要时插入新的复习任务
  → 重新平衡每日容量
  → 搭子解释“为什么计划发生变化”
```

## 4. 建议技术架构

### 4.1 架构原则

1. **单体优先、边界清晰**：继续使用一个 Node.js/TypeScript 应用，不拆微服务。
2. **领域逻辑与文件 I/O 分离**：计划、评分、掌握度计算保持纯函数；文件、模型和搜索通过适配器接入。
3. **工作流负责串联，Agent 负责决策**：不要让 `cli.ts` 继续承担全部编排。
4. **结构化输出必须验证**：所有外部搜索结果和 LLM JSON 先通过运行时 Schema，再进入状态文件。
5. **稳定标识和溯源优先**：材料、切片、概念、题目、任务都使用不会因重新运行而碰撞的 ID。
6. **人类确认是正式状态**：来源确认和首版计划确认写入状态与审计日志。
7. **搭子只通过应用服务读取业务状态**：不直接散落读取各个 JSON 文件。

### 4.2 逻辑分层

```text
Interfaces
  CLI（保留） / Local Web UI（逐步新增） / Obsidian Export（可选）
        ↓
Workflows / Application
  bootstrap_exam / research_exam / build_knowledge
  generate_plan / run_daily_study / grade_and_adapt
        ↓
Domain
  ExamProject / Source / Material / Chunk / Concept
  StudyPlan / Task / Quiz / LearnerModel / BuddyState
        ↓
Ports
  SearchProvider / LLMClient / MaterialParser / Repositories / Clock
        ↓
Infrastructure Adapters
  Web Search / OpenAI-compatible LLM / PDF+Markdown / Local Files / Event Log
```

### 4.4 领域对象

#### ExamProject

```ts
interface ExamProject {
  id: string;
  name: string;
  examDate: string;
  target?: string;
  learnerProfile: {
    baseline: 'beginner' | 'intermediate' | 'advanced';
    dailyMinutes: number;
    unavailableDates: string[];
  };
  status:
    | 'draft'
    | 'researched'
    | 'sources_approved'
    | 'materials_ready'
    | 'planned'
    | 'active'
    | 'completed';
  schemaVersion: number;
}
```

#### SourceRecord

至少包含：URL、标题、来源类型、发布者、采集时间、考试版本、摘要、可信度理由、内容哈希、是否经用户确认。来源类型建议固定为 `official | community | commercial | user_file`。

#### LearnerModel

不要只存一个 `mastery` 浮点数。至少保留：概念掌握度、最近练习时间、证据数量、连续正确次数、错误类型分布、下次复习日期和掌握度变更历史。

#### BuddyState

至少包含：角色 ID、称呼、互动偏好、关系阶段、近期承诺、最近重要学习事件。角色不得直接篡改计划；它通过工作流请求解释、提醒或调整。

### 4.5 状态和事件

下一阶段采用“**状态快照 + 审计事件**”，不急于实现完整 Event Sourcing：

- JSON 文件保存当前可读状态；
- JSONL 记录发生过的业务事件；
- 每个事件增加 `schemaVersion`、`correlationId`、`examProjectId`；
- 写入采用临时文件 + 原子替换，避免中途损坏；
- 重要状态保留版本或调整日志；
- 等 reducer 和 replay 测试完成后，再将事件日志升级为事实来源。

建议的工作流事件包括：

- `exam_project_created`
- `research_completed`
- `sources_approved`
- `material_imported`
- `knowledge_built`
- `plan_proposed`
- `plan_approved`
- `task_completed` / `task_skipped`
- `quiz_graded`
- `learner_model_updated`
- `plan_adjusted`
- `buddy_intervention_generated`

### 4.6 搜索与 LLM 边界

考试调研必须先搜索、后综合，不能让 LLM 凭记忆生成“备考经验”。

```text
SearchProvider 返回原始结果
  → SourceNormalizer 去重与分类
  → ExamResearcher 基于来源生成结构化结论
  → CitationValidator 检查每条关键结论是否有来源
  → 用户批准来源
  → 才能进入 MaterialCollector
```

LLM 只承担难以规则化的任务：经验提炼、概念抽取、出题和搭子表达。下载、哈希、去重、排程容量、评分和状态迁移由 TypeScript 控制。

### 4.7 界面策略

#### 近期

- CLI 用于开发、自动化和完整功能覆盖。
- 生成清晰的 Markdown/JSON 产物。
- Obsidian 保持可直接浏览，但不再定义为唯一默认界面。

#### 中期

新增本地 Web UI，最少包含：

- 考试建档向导；
- 调研来源确认页；
- 今日任务页；
- 答题与反馈页；
- 搭子角色与对话区；
- 计划和掌握度视图。

先做静态立绘、状态切换和文本对话，不做 3D/实时动画。这样能验证“实体感”是否真的提高留存，再决定投入语音和动画。

## 5. 分阶段开发路线

估算以一名熟悉项目的开发者为基准，总计约 24～36 个工程日。每个阶段都必须独立可验收。

### Phase 0：稳定现有底座（2～3 日，P0）

目标：避免在不可靠数据上增加新功能。

- 修复多材料导入时 `chunk_001` 等文件互相覆盖。
- 处理无标题正文、空切片和重复标题。
- 为 Material/Chunk 使用稳定 ID 与内容哈希。
- 为概念依赖增加循环检测。
- 移除计划最多 14 天的限制，补充非法考试日期和每日时长校验。
- 将薄弱点画像改为累积更新，不再被当天结果覆盖。
- 将 `cli.ts` 中 grade 后的编排抽成 `grade_and_adapt` 工作流。
- 为以上缺陷增加回归测试。

验收：连续导入 3 份资料不会覆盖；可生成 60 天计划；连续 3 次测验后历史薄弱点仍可追踪。

### Phase 1：考试建档与备考调研（4～6 日，P0）

目标：把产品入口从“上传文件”升级为“告诉我你要考什么”。

- 新增 `ExamProject`、状态机和持久化。
- 新增 `exam create` 或等价建档入口。
- 定义 `SearchProvider` 接口与首个搜索适配器。
- 实现官方/社区/商业/用户文件四类来源。
- 实现去重、采集时间、考纲版本和可信度理由。
- 生成考试画像、经验共识、争议点、资料推荐。
- 新增用户来源确认步骤。
- 所有关键结论带来源引用；无证据时明确标记“证据不足”。

验收：输入一个真实考试名称后，输出至少一条官方来源、若干经验来源和带引用的资料建议；未经确认的来源不会进入知识库。

### Phase 2：可靠资料与知识构建（4～6 日，P0）

目标：让多来源内容可靠进入学习系统。

- 支持经批准 URL 的正文抓取。
- 建立 `materials/index.json` 和 `chunks/index.json`。
- 支持 token/字符上限下的分段和最小片段合并。
- 保留章节路径、来源定位和材料版本。
- 概念抽取分批执行，再做合并、去重和依赖校验。
- 每个概念保存证据切片；无法回链的概念不进入正式图谱。
- 增加人工确认或修正知识结构的接口。

验收：官方大纲、教材和经验资料可以分类入库；概念和题目均能回链到来源切片。

### Phase 3：完整计划与每日执行（4～5 日，P0）

目标：计划覆盖整个考试周期，并能处理现实中的跳过和变化。

- 根据概念数量、难度、权重和用户时间估算任务容量。
- 生成完整周期，包含学习、复习、测验、缓冲和冲刺阶段。
- 增加首版计划确认。
- 支持不可学习日期、跳过、完成和未完成任务顺延。
- 计划调整能够插入新复习任务，而不只是给既有任务加时。
- 每次调整保留原因和前后差异。

验收：60 天计划每天不超过容量；跳过一天后任务能合理迁移；薄弱点可插入未来计划并解释原因。

### Phase 4：证据化测验与自适应（4～6 日，P0/P1）

目标：出题、反馈和计划调整真正围绕用户当天学习证据运行。

- 题目只覆盖今日知识点、到期复习点和少量历史薄弱点。
- 支持单选和多选；题数、难度和比例可配置。
- 每道题强制关联概念与来源切片。
- 对答案、选项数、重复题、来源关联做结构校验。
- 累积错误类型、连续正确次数和掌握度历史。
- 将“错误类型”从固定值升级为规则 + 用户确认，避免 LLM 随意归因。
- 建立策略指标：计划完成率、复习后正确率、知识保持率和题目废弃率。

验收：连续完成至少 3 个日循环后，系统能展示为什么某概念被判为薄弱、为什么它被重新安排。

### Phase 5：搭子产品化与本地 Web UI（5～8 日，P1）

目标：验证“有实体感的陪伴”是否提升执行，而不是只增加装饰。

- 搭子状态从散落 JSON 读取改为统一应用上下文。
- 引入 `BuddyState` 和跨会话重要记忆。
- 设计关键干预时刻：建档、计划确认、开始任务、跳过、连续完成、低分、进步、临考。
- 搭子必须能解释系统决策，不能伪造学习数据。
- 新增本地 Web UI 最小闭环。
- 提供至少一名男性、一名女性和一个卡通吉祥物的静态立绘状态。
- 支持用户控制提醒强度、称呼和情感风格。
- CLI 保持同等业务能力，UI 只调用工作流，不复制规则。

验收：非技术用户可以在 Web UI 中完成建档、来源确认、今日学习、答题和查看调整；角色在关键节点保持人设且引用真实学习状态。

### Phase 6：真实场景验证与发布加固（3～5 日，P1）

目标：从“模块测试通过”升级到“真实考试流程可信”。

- 选择一个真实考试做金样本。
- 使用真实搜索和真实 LLM 跑完整流程。
- 建立调研引用、概念抽取和题目质量评估集。
- 增加端到端测试、失败恢复和幂等测试。
- 记录模型、Prompt 版本、耗时和成本。
- 修订 README、PRD 和演示材料，删除未实现能力的宣传。

验收：新工作区从建档到三天学习循环可重复跑通；中途失败后不会重复导入或破坏状态；文档描述与实现一致。

## 6. 代码级优先任务映射

| 现有位置 | 首要改动 | 原因 |
|---|---|---|
| `src/agents/chunker.ts` | 稳定 ID、索引、无标题正文、避免覆盖 | 当前多材料路径不安全 |
| `src/agents/concept_mapper.ts` | 分批抽取、循环检测、证据回链 | 当前截断且存在循环风险 |
| `src/agents/planner.ts` | 完整周期、容量规划、日期校验 | 当前最多 14 天 |
| `src/agents/mistake_analyzer.ts` | 累积画像、错误类型证据 | 当前覆盖历史且固定归因 |
| `src/agents/plan_adjuster.ts` | 插入任务、差异与原因 | 当前只为既有复习任务加时 |
| `src/agents/quiz_generator.ts` | 今日范围、多选、来源回链 | 当前从全部概念出单选题 |
| `src/agents/study_buddy.ts` | BuddyState、重要记忆、干预策略 | 当前主要是角色 Prompt |
| `src/cli.ts` | 移出业务编排，调用工作流 | 当前职责过多、修改易连锁 |
| `src/core/event_log.ts` | 事件版本、关联 ID、项目 ID | 当前只能简单追加和读取 |
| `src/core/paths.ts` | ExamProject 感知的 WorkspacePaths | 当前全局单工作区限制扩展 |

## 7. 测试策略

### 单元测试

- 领域纯函数：排程、掌握度、任务迁移、来源评分、状态迁移。
- Schema 验证：搜索结果、LLM 输出、状态文件兼容。
- 边界情况：过期考试、零时间、循环依赖、重复资料、空正文。

### 契约测试

- SearchProvider 返回统一 SourceRecord。
- MaterialParser 对 PDF/Markdown/URL 返回统一 MaterialDocument。
- LLMClient 的 JSON 输出经过同一验证层。

### 集成测试

- 多材料导入到知识图谱。
- 计划生成到任务执行。
- 出题、批改、掌握度更新和计划调整。
- 事件和状态同时写入失败时的恢复。

### 端到端金样本

固定一门考试、固定若干来源、固定模型输出快照，验证：

- 官方信息没有被经验帖覆盖；
- 概念均有来源；
- 计划容量合理；
- 题目只考允许范围；
- 三轮学习后薄弱点和计划变化可解释。

## 8. 产品指标

第一阶段不要以“生成了多少内容”为成功指标。建议记录：

- 首次建档到首份可用计划的完成率和耗时；
- 来源确认率、被用户删除的来源比例；
- 每日任务完成率和连续学习天数；
- 测验题被用户废弃/纠错的比例；
- 同一知识点复习后的正确率变化；
- 动态调整后次日任务完成率；
- 搭子干预开启/关闭情况下的任务完成差异。

## 9. 发布门槛

满足以下条件才可以把下一阶段称为“完整备考 Agent MVP”：

- [ ] 可以从考试名称开始，而不是必须由用户先准备资料。
- [ ] 关键考试事实和经验结论有来源引用。
- [ ] 至少支持 3 份资料且不会覆盖或混淆来源。
- [ ] 能生成 30～60 天完整计划。
- [ ] 题目来自今日内容并回链资料。
- [ ] 错题和薄弱点跨多天累积。
- [ ] 动态调整能够插入和迁移任务。
- [ ] 搭子读取真实学习状态并能解释关键变化。
- [ ] 真实搜索 + 真实 LLM 至少完成一个三日金样本流程。
- [ ] PRD、README、演示文案与实现保持一致。

## 10. 开发决策记录

以下为本计划采用的默认决策，如产品方向改变，应先更新本节再改代码：

1. 单用户、单考试项目优先，但数据模型包含 `examProjectId`。
2. 本地文件继续作为默认存储，不引入数据库。
3. 事件日志近期定位为审计日志，不宣称完整 Replay。
4. CLI 是完整能力入口；本地 Web UI 是下一阶段主用户界面。
5. Obsidian 是可选浏览/导出能力，不是拟人化体验的唯一载体。
6. 先做用户自适应，不做不可控的系统自修改。
7. 先验证静态立绘 + 状态化文本陪伴，再决定语音、动画和 3D 投入。
